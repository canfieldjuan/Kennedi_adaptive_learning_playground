# v0.3.3 Bear Cafe Child-Started Play Review Contract

## Before Code

### Root Cause

Bear Cafe now starts from the child home grid, so parent review must explain the child-started order flow. The Bear Cafe runtime already emits `completed` delivery events with prompt text, selected tray, correct order, response time, hint state, and outcome. The parent Recent Attempts formatter excludes those delivered-order events, so the parent can see a completed activity summary but can miss the Cafe-specific evidence that explains what actually happened in the shift.

### Correct Fix Must Touch

- `src/core/parent-review-format.ts` to include Bear Cafe delivered-order completion events in Recent Attempts without admitting ordinary completion bookkeeping events.
- `tests/contract/parent-review-format.test.ts` to prove Bear Cafe delivered orders show friendly title, skills, prompt, selected answer, correct answer, hint usage, response time, and outcome, while ordinary TapChoice/Coloring success-plus-completion pairs do not duplicate recent attempts.
- `docs/work-contracts/v0.3.3-bear-cafe-play-review.md` to record the before-code contract and final audit.
- `CURRENT_STATUS_AND_ROADMAP.md` and `MVP_BASELINE.md` to update the current product state.
- `package.json` and `package-lock.json` to bump the app version if verification passes.

### Must Not Change

- Do not change the Bear Cafe runtime, activity JSON, activity chain, child home grid, Video Vault direct route, event schema, activity schema, storage schema, export/reset behavior, mastery engine, transfer coverage engine, recommendation engine, parent approval semantics, or Vercel config.
- Do not add new game modules, new activities, `Nature Camera Safari`, backend services, auth, cloud sync, AI tutor chat, open web content, rewards, streaks, leaderboards, public sharing, or automatic adaptive routing.
- Do not weaken safety, parent approval, transfer truth, mastery, or local-first data rules.

## Contract Amendments

- PR review found that globally including `completed` would duplicate ordinary TapChoice and Coloring successes because those runtimes emit `correct` followed by immediate `completed` for the same action. The fix was narrowed to Bear Cafe delivered orders identified by `activity_id` plus `metadata.event_name === "order_delivered"`.
- A follow-up PR review found that Bear Cafe itself can still duplicate a successful order because the runtime emits a `correct` `tray_checked` event before the later `completed` `order_delivered` event. The fix must collapse only that matching Bear Cafe success pair so the delivered order is shown once while undelivered tray-check successes and earlier incorrect/hint evidence remain visible.
- A second follow-up PR review found that a session-wide delivered-order key can suppress a later identical tray-check that has not been delivered. The fix was narrowed again to chronological pairing: only the specific tray-check event immediately paired with a later matching delivery is hidden.
- A third follow-up review left the original duplicate-row P2 active because Bear Cafe tray-check successes were still admitted through the generic `correct` allowlist. The fix made Bear Cafe tray-check success handling explicit before the generic allowlist and added the exact six-row crowd-out regression.
