# v0.3.30 Number Train — curriculum and evidence alignment (arc slice 5)

Fifth slice of the Number Train build arc (issue #43): the Math skills the game
actually exercises become distinct curriculum skills with honest per-skill
evidence, and the Bear Cafe-flavored counting rungs become general Math rungs.

## Before Code

### Root Cause (slice)

The curriculum conflates what Number Train exercises. Every Number Train round
records evidence under `counting` alone, though count rounds also exercise
numeral recognition, load rounds exercise quantity construction, and missing-
station rounds exercise number order — three capabilities with no curriculum
skills at all. Counting's rung labels are Bear Cafe-flavored ("Counts small
pretend-play sets", "Counts in combined orders") rather than general Math
rungs. The per-skill outcome machinery (`skill_outcomes`, added in #39) exists
but Number Train does not use it.

### Correct Fix Must Touch

- `src/content/curriculum/curriculum.v1.json`:
  - **counting** — relabel level 1 → "Counts structured quantities accurately"
    and level 2 → "Counts grouped quantities across tens" (general Math rungs;
    **bands, evidence requirements, and review policy unchanged**, so existing
    stored evidence and promotion state are untouched — labels are
    presentational); add `number-train` to `activity_contexts`; `unlocks`
    gains `quantity_construction` and `number_sequence` (no consumer outside
    the graph module — graph semantics only).
  - **new `numeral_recognition`** — reading/matching written numerals
    (no prerequisites: symbol recognition enters in parallel with counting).
  - **new `quantity_construction`** — building a requested quantity
    (prerequisite `counting`).
  - **new `number_sequence`** — number order / missing numbers
    (prerequisite `counting`).
  - Each new skill: contiguous levels 0–2 with non-overlapping bands
    (0–1 / 2–3 / 4–5), planned transfer contexts including the envelope's
    `same_format_new_examples`, evidence requirements and review policy in
    line with the existing math skills, `activity_contexts: ["number-train"]`.
  - **subitizing — unchanged**: it stays bounded to the small-set legacy
    activities (quantity three is genuinely subitizable); Number Train
    continues to emit **no** subitizing evidence.
- `src/content/activities/number-train.json` — `skill_ids` and
  `transfer.skill_ids` become the four skills the session exercises
  (`counting`, `numeral_recognition`, `quantity_construction`,
  `number_sequence`).
- `src/modules/number-train/NumberTrainActivity.ts` — per-round skill
  attribution: `roundSkillIds` maps count → [counting, numeral_recognition],
  load → [counting, quantity_construction], missing station →
  [number_sequence, numeral_recognition]; every event carries those
  `skill_ids` plus matching `skill_outcomes` entries (the #39 extension the
  progress/evidence engines already consume via `getSkillOutcome`).
- Tests:
  - `tests/modules/number-train.test.ts` — per-round skill/skill_outcomes
    assertions for all three kinds (and still never subitizing); envelope
    assertion updated to the four skills; a curriculum check that difficulty 1
    maps to the entry level for **all four** skills (the PR #44 invariant
    generalized).
  - `tests/contract/curriculum-graph.test.ts` and
    `tests/contract/parent-game-launch.test.ts` — the two assertions pinning
    the old Bear Cafe-flavored counting labels updated to the general labels
    (intentional; the tests pin presentational strings).
- `package.json` / `package-lock.json` — version bump to `0.3.30`.
- Docs note in `MVP_BASELINE.md`.

### Parent-readable summaries (decision documented)

Number Train events flow through the parent Recent Attempts **default**
rendering: short prompt text, selected vs correct answer, hint state, response
time, and the round metadata (`round_type`, `round_index/total`,
`target_quantity`). Per-tap seat interactions were already excluded at the
source (no events until Check), so the parent record is one meaningful row per
evaluated answer — no bespoke formatter needed this slice. Bear Cafe's custom
formatter pattern remains available if richer summaries are wanted later.

### Coverage limitation (explicit, per the arc spec)

Only the **entry band** (difficulty 0–1) of the four skills has a producing
approved activity today (`number-train`, difficulty 1). The level-1/2 bands
(difficulty 2–5) are declared as rungs but have no producing activity until
authored higher-difficulty session configs land (the 21–50 range extension
seam). This is the same app-wide band-coverage pattern tracked in issue #35 —
documented here rather than hidden, and preferable to declaring single-level
skills that would need graph surgery later.

### Explicit Non-Scope / Must Not Change

- Gameplay, round composition, journey/arrival/replay, hints — untouched.
- `subitizing` skill definition and the legacy Math activities' envelopes —
  untouched (their small-set counting+subitizing evidence is honest).
- Counting bands/evidence/review (labels only), event contract (the
  `skill_outcomes` field exists since #39), storage keys, dispatch, home
  route, other runtimes — untouched.
- No adaptive routing, no new activities, no reward systems.

## Contract Amendments

Two Codex review findings on PR #51 amended the slice (both confirmed against
the code):

1. **Hint attribution narrowed to the hinted skills.** The generic per-skill
   mapping recorded `hint_used` for every round skill, so a count-along hint
   polluted `numeral_recognition`'s hint rate. Hint events now attach only to
   the skills the hint teaches (count → counting; load → counting +
   quantity_construction; sequence → number_sequence), and post-hint events
   carry `metadata.hinted_skill_ids` (the CSV shape
   `skill-outcomes.ts getHintedSkillIds` parses) so downstream per-skill hint
   attribution stays precise. The highlighted numeral is target marking, not
   numeral-recognition teaching.
2. **`src/core/parent-interpretation.ts` touched (was Must-Not-Change).**
   The new prerequisite gates exposed a pre-existing engine gap: Parent
   Guidance called `evaluateSkillMastery` without `prerequisite_statuses`, so
   the mastery engine treated every prerequisite as unmet and gated skills
   (ours, and the pre-existing subitizing/letter_sound_match/blending/
   word_building) could never leave introduced/practicing in the parent lane.
   Fixed at the root: `buildParentSkillInterpretations` now evaluates mastery
   with a memoized recursive pass that feeds each skill its prerequisites'
   evaluated statuses (cycle-guarded; the graph is validated acyclic). A
   regression test proves the gate lifts when the prerequisite reaches
   likely_mastered — and fails without the fix (verified by stash-run).
