# v0.3.61 Bear Cafe Ownership Completion Change Contract

Status: active
Owner: Game Builder session

## Root Cause

Bear Cafe currently ends with a generic delivery payoff and then discards the
child's completed order. The game evaluates a tray, but it does not create a
durable completion object that includes an unscored ownership choice, preserve
that exact object through delivery, or give the child a no-labor way to revisit
past work. Free Make also routes expressive choices through assessed event
semantics, so open-ended ownership can be misread as skill evidence.

The root defect is not missing decoration or missing animation in isolation.
The game lacks one authoritative, bounded, persistent representation of what
the child made and personalized. Without that representation, immediate payoff,
revisit, export, clearing, evidence boundaries, and exact visual continuity
cannot agree.

## Correct Fix Standard

A correct fix must:

1. Create one immutable Bear Cafe completion record containing the actual
   caller, ordered food quantities, applied food color/decoration when present,
   chosen bag color, chosen seal, activity identity/version, child/session
   identity, and completion timestamp.
2. Add one bounded packaging action after cooking and before delivery. The
   child chooses one of four always-available bag colors and one of three
   always-available seals. These choices are expressive, deterministic,
   unscored, and never gated by currency, streaks, randomness, or performance.
3. Render the same completion record through packaging confirmation, delivery,
   customer handoff, the child-controlled finish screen, and later revisit.
   No generic substitute may replace the child's actual bag, seal, caller, or
   food contents.
4. Persist valid records locally under a dedicated Bear Cafe history key,
   retain the newest 48, reject malformed/unknown records on read, make append
   idempotent by completion id, include the records in local export, and clear
   them through the existing parent clear-data action.
5. Add a child-accessible order wall inside Bear Cafe, with no new route or
   earned access. It must show newest-first pages of at most six orders, allow
   exact-detail inspection, preserve a clear return path, and expose no
   child-mode delete action.
6. Keep the final exact result visible until the child chooses another order,
   opens the order wall, or goes home. Timers may animate handoff but may not
   erase the completion object or start another activity.
7. Preserve all assessed activity answers, difficulty, transfer metadata,
   approval state, and assessed event outcomes. Free Make must emit exactly one
   completion event for a completed order, with `skill_outcomes: []`, and must
   not create vocabulary or other mastery evidence from ownership choices.
8. Use original, locally stored vector artwork with editable source and
   documented provenance. Runtime assets must have no remote references,
   authoring-tool dependency, third-party library content, inaccessible child
   instructions, or proprietary fonts.
9. Preserve preschool interaction safety: large targets, tap-first controls,
   no failure screen, no time pressure, no harsh sound, no red-X feedback, and
   full operation with speech, sound, or motion disabled.

## Completion Record Contract

The stored record must use a game-specific schema with these boundaries:

- `schema_version` is exactly `1` and `game` is exactly
  `kennedis-orders`.
- Identity strings are non-empty and bounded.
- Caller, food, food-color, food-decoration, bag-color, and seal ids come from
  explicit local allowlists.
- Food lines are unique, number from 1 through 6, each count is an integer from
  1 through 5, and total quantity is at most 12.
- `completed_at` is a valid bounded timestamp.
- Unknown or extra input fields are never trusted as runtime state; valid known
  fields are reconstructed into a fresh object.
- Missing storage is an empty history. No migration is required.
- Duplicate completion ids do not append a second record.
- Reads return at most the newest 48 valid records.

The game receives only the narrow operations needed to list and append these
records. It must not receive or duplicate the full progress system.

## Interaction Contract

The bounded child flow is:

`answer -> make/fix -> cook -> package -> deliver -> handoff -> admire`

Packaging starts with `pink` and `heart` selected so the child can continue
without extra fine-motor work. Available bag colors are pink, yellow, orange,
and blue. Available seals are heart, star, and bubbles. A package preview must
update immediately and the confirmed exact object must remain unchanged for the
rest of the payoff.

The order wall is reachable from the Bear Cafe phone screen when history exists
and from the completion screen. It shows six records per page: a 3-by-2 layout
where space permits and a 2-by-3 layout on a narrow child viewport. Opening a
card shows the exact caller, bag, seal, and food contents. Revisit emits no
activity attempt and requires no repeated lesson.

The child-controlled finish screen must not auto-advance or auto-clear. One
core learning action, at most two ownership choices, and one payoff keeps the
flow bounded.

## Evidence Contract

- Assessed Bear Cafe variants keep their existing meaningful `correct`,
  `incorrect`, `hint_used`, and `completed` emissions and skill outcomes.
- Bag color, seal selection, wall browsing, and detail inspection emit no
  learning-attempt events.
- Free Make emits no intermediate assessed selection/check events. Delivery
  emits one existing-shape `completed` event with `skill_outcomes: []`.
- Parent formatting and session skill summaries must use effective skill
  outcomes, so a Free Make completion is shown as unscored ownership rather
  than vocabulary evidence.
- Persistence of the completion object is separate from the existing event log
  and must not create a second mastery/progress system.

  hints, difficulty, transfer context/strength, skill graph, mastery,
  recommendations, or parent approval.
- Child routes, home cards, parent navigation, accounts, auth, backend, cloud
  sync, or network behavior.
- External links, open-web content, AI tutor behavior, real camera access,
  social features, ads, streaks, leaderboards, currency, random rewards, or
  reward economies.
- Dependencies, public APIs unrelated to the narrow injected history port,
  storage migrations, or unrelated formatting/refactors.
- Existing Bear Cafe caller, food, color, and decoration content except where
  the immutable completion record must faithfully reference it.

The shared detached checkout, every sibling worktree, every other branch/PR,
and the untracked shared workflow file are outside this lane.

## Assumptions And Blockers

- The existing storage adapter can accept one additive local key and bounded
  repository without a migration or export-version bump.
- Existing event semantics support an explicit empty `skill_outcomes` array.
- The current Bear Cafe view-state machine can accept packaging, wall, detail,
  and child-controlled completion states without a route change.
- Original vector assets can be authored as standards-based SVG in Inkscape;
  no third-party or AI-generated artwork is required.
- If inspection contradicts any assumption, implementation pauses at the
  smallest blocker instead of expanding curriculum, schemas, or infrastructure.
