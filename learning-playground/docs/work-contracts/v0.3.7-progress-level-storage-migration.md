# v0.3.7 Progress Level Storage Migration

## Before Code

### Root Cause

v0.3.6 made curriculum rungs load-bearing during progress rebuilds, but a stale
saved `lp_child_progress_profile` can still be read before any new activity
event triggers a rebuild. Parent Panel and export both call `getProgressProfile`,
so stale raw `current_level` values can remain parent-visible and export-visible
until the child plays again.

### Correct Fix Must Touch

- `src/core/progress.ts`
  - Expose a normalization helper that clamps known skill levels to their
    declared curriculum ladder without rebuilding from events.
- `src/core/storage.ts`
  - Normalize progress profiles on read.
  - Save the normalized profile back when a stale stored level is corrected.
- Focused progress/storage/export test proving stale stored levels are corrected
  before parent review/export.
- Version/baseline/current-status docs for v0.3.7.

### Must Not Change

- Do not change storage keys.
- Do not change event schema, activity schema, mastery rules, transfer coverage,
  recommendation logic, parent approval, Bear Cafe runtime, activity chain, home
  grid, export section names, reset behavior, backend/auth/sync posture,
  rewards, streaks, or automatic routing.

## Cold Diff Audit

### Gaps

- change without contract trace: none.
- contract requirement not delivered: none.

### Change By Change Reconstruction

- `progress.ts` adds `normalizeProgressProfileLevels`, which clamps known skill
  `current_level` values to the curriculum ladder and reports whether anything
  changed.
- `storage.ts` calls the normalization helper inside `getProgressProfile` and
  saves the corrected profile when needed.
- `progress-tracking.test.ts` seeds a stale saved profile at counting level `5`
  and proves `getProgressProfile`, the stored JSON, and export all use the
  corrected curriculum max level.
- `package.json` / `package-lock.json` bump to `0.3.7`.
- `MVP_BASELINE.md` and `CURRENT_STATUS_AND_ROADMAP.md` record the read-time
  storage migration.

### Contract Traceability

- Stale stored profile risk is closed at the read boundary used by both Parent
  Panel and export.
- Existing storage key remains unchanged.
- Unknown or unsupported progression behavior is not expanded into a schema
  migration.
- Mastery, transfer, schemas, parent approval, reset behavior, and Bear Cafe
  runtime remain outside this change.
