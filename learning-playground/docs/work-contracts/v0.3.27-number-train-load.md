# v0.3.27 Number Train — Load the Train (arc slice 3)

Third slice of the Number Train build arc (issue #43): **quantity
construction**. The trip gains "Load the Train" rounds — the game asks for a
target quantity and the child builds it by seating passengers, checking the
train when ready. Evaluated on Check, never per tap.

## Before Code

### Root Cause (slice)

The session can only ask the child to *read* a quantity (count and pick a
numeral). It cannot ask the child to *construct* one — there is no interaction
for building a requested quantity, no evaluation of a constructed answer, no
add/remove-before-submit flow, and no structural "how many more" hinting. The
trip therefore exercises only recognition, not production.

### Correct Fix Must Touch

- `number-train.types.ts` — add
  `LoadTrainRound { kind: 'load_train'; target; prompt }`; the round union
  becomes `CountTrainRound | LoadTrainRound`.
- `round-plan.ts` — `validatePlan` gains load-round rules (integer target,
  `1..max_quantity`, non-empty prompt); `buildSessionPlan` emits a
  deterministic mixed composition: for the standard six-round trip, rounds 4
  and 5 are Load-the-Train (build the ramp quantity), the rest stay
  Count-the-Train. Same seed ⇒ same mixed plan; all existing generator rules
  hold (ramp, bounds, fail-closed validation).
- `NumberTrainActivity.ts` — `renderRound` dispatches on `round.kind`:
  - **Load round**: the train renders with enough cars for the target but
    **empty seats**; prompt "Put N passengers on the train." Seating is
    direct AND large-control based: tapping an empty seat seats a passenger,
    tapping a seated passenger removes them (remove-before-submit), and
    large **Add passenger** / **Remove** / **Check** child-buttons do the
    same for safe tap targets. Seat taps emit **no events**; pressing
    **Check** evaluates the constructed count against the target — correct →
    the normal round-complete path (`correct` event with the constructed
    answer, journey fill, Next station / arrival); wrong → `incorrect` event
    (selected = the constructed count) with short supportive feedback ("N
    more…" / "too many…"), and after two wrong Checks one bounded structural
    hint: speak how many more are needed and highlight the next empty seat —
    **never auto-fill**. `attempt_number` counts Checks, not taps.
  - Count rounds unchanged.
- `child-ui.css` — tappable-seat affordance for load rounds, next-seat hint
  highlight, load-controls row; any animation added to the reduced-motion
  list.
- `tests/modules/number-train.test.ts` — mixed-plan determinism and validity
  (rounds 4–5 are `load_train` across seeds; load targets within bounds);
  load-round behavior: renders capacity cars with zero occupied; seat tap
  seats / re-tap removes; Add/Remove controls work; **no events until
  Check**; wrong Check → `incorrect` with the constructed count +
  hint-after-two-wrong highlighting the next empty seat without auto-filling;
  correct Check → `correct` + Next station; a full mixed trip still emits
  `completed` exactly once; validation rejects a load target above max.
- `package.json` / `package-lock.json` — version bump to `0.3.27`.
- Docs note in `MVP_BASELINE.md`.

### Explicit Non-Scope / Must Not Change

- **Evidence stays `counting`** this slice: constructing a quantity is
  counting production, recorded under the existing skill; the distinct
  `quantity_construction` skill, per-skill outcomes, and curriculum split are
  slice 5 (documented limitation, not hidden).
- Missing Station (slice 4), curriculum changes, difficulty band changes —
  out of scope. The envelope (seed/round_count/max/difficulty) is unchanged.
- Count-round behavior, journey/arrival/replay semantics, event contract,
  dispatch, home route, protected modules (`tap-choice/`, phonics, Bear Cafe,
  `src/core/`, curriculum, schema) — untouched.
- No per-tap Math attempts, no timers, no reward loops, no auto-fill hints.
