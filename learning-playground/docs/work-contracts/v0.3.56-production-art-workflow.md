# v0.3.56 Production Art Workflow and Ownership

Documentation-only process slice. The package version remains v0.3.56 under
the repository's no-bump convention for art-direction contracts that do not
change runtime behavior or assets.

## Before Code

### Root Cause

The repository has a visual-environment standard but no canonical production
art ownership workflow. Existing contracts also require inline SVG in ways
that would reject editable source files and optimized local exports from a
professional vector editor. Without one governing contract and provenance
ledger, a future graphics pass could ship visually polished assets that are
legally unclear, difficult to revise, dependent on an authoring platform, or
inconsistent with the local-first runtime boundary.

### Correct Fix Must Touch

- `docs/contracts/art-production-assets.contract.md` (new): define approved
  authoring tools, ownership categories, Canva and third-party restrictions,
  editable-source and provenance requirements, AI/character rules, delivery
  artifacts, export standards, implementation boundaries, and one-scene look
  approval before broad rollout.
- `docs/art/asset-provenance.md` (new): establish the canonical production
  asset ledger, required fields, review states, entry template, and the rule
  that undocumented assets are not production-ready.
- `design-source/README.md` (new): establish one editable-source root and map
  its game folders to existing production asset locations without adding art.
- `docs/contracts/game-environment.contract.md`: replace the categorical
  inline-SVG-only rule with local, project-owned art governed by the new
  production contract while retaining scene, hierarchy, safety, mobile, and
  performance rules.
- `docs/contracts/art-studio.contract.md` and
  `docs/contracts/story-stage.contract.md`: remove only their conflicting
  inline-SVG-only wording and point visual assets to the production contract.
- This work contract: reconstruct and audit the final documentation diff.

### Must Not Change

- No production art, editable art source, exported asset, screenshot, contact
  sheet, asset loader, application code, CSS, content, test, route, dependency,
  package version, or lockfile change.
- No gameplay, correct answer, evaluation rule, event timing, evidence,
  mastery, transfer, curriculum, difficulty, parent behavior, persistence, or
  completion-semantic change.
- No retrospective rewrite of historical work contracts or claims about the
  provenance of existing assets that the repository cannot prove.
- No Canva library asset, Figma Community resource, stock media, third-party
  font, AI-generated image, external link in child mode, hosted design, runtime
  account requirement, or design-platform dependency.
- Do not edit or stage the unrelated `.github/workflows/pr-review-wake.yml` in
  the shared checkout, or modify another session's branch, worktree, PR, or
  persisted state.

### Assumptions and Blockers

- This slice creates policy and templates only. Existing art may continue to
  run, but a later graphics PR must establish provenance and editable source
  for every asset family it changes or promotes as production art.
- A substantial proprietary source file may be committed later when real art
  exists; this slice creates only the source-tree convention.
- No blocker requires a runtime, schema, or package-version change.
