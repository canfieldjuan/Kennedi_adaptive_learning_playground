# v0.3.33 Video Vault Evidence Boundary

## Before Code

### Code Read Cold

- The Video Vault already has a local-only path: its runtime keeps only items
  with `source: local`, explicit parent approval, a local `/assets/video...`
  path, and no external thumbnail URL.
- The app imports one repository JSON manifest and casts it to
  `VideoVaultManifest`. The manifest is empty, so no real item currently passes
  through the runtime or the item-level safety assertions.
- The runtime records an ended video as `outcome: completed`, `input_type:
  video`, and attaches the activity's `vocabulary` skill id.
- `completed` is not a counted skill outcome. Only correct, incorrect, and
  abandoned outcomes affect counted attempts. Consequently, a video completion
  can create completion evidence but cannot create accuracy, successful
  transfer context, or vocabulary mastery; mastery remains `not_started` when
  no counted attempt exists.
- The runtime does not verify that the activity's requested manifest id matches
  the supplied manifest. It does not validate manifest identity, duplicate item
  ids/paths, duration bounds, supported media type/extension, or an explicit
  evidence role. Invalid items are only partially filtered and silently hidden.
- The roadmap says the local-only content path still needs deciding, but the
  code has already selected repository-bundled local assets. Parent-imported
  media has no blob storage or IndexedDB path.

### Root Cause

The Video Vault's safety and evidence boundaries are implicit and only partly
enforced. Local repository media is already the v1 intake decision, and video
completion is already non-mastery exposure, but neither fact is represented by
a validated manifest contract or machine-readable event metadata. Adding media
now would rely on a TypeScript cast, vacuous empty-manifest tests, and silent
filtering rather than a durable policy.

### Correct Fix Must Touch

- Add `docs/contracts/video-vault-evidence.contract.md` declaring
  repository-bundled local assets as v1, parent imports deferred until durable
  blob storage exists, and video completion as exposure rather than a skill
  response.
- Extend the Video Vault manifest types with an explicit repo-bundled intake
  mode, unsupported parent-import flag, supported MIME type, and
  `exposure_only` evidence role.
- Add a pure manifest validator that checks manifest identity/version/title,
  activity-manifest id agreement, unique item ids and paths, local paths,
  supported MIME/extension agreement, finite bounded duration, parent approval,
  exposure-only role, local thumbnails, and no autoplay/loop chaining.
- Make the runtime render only validator-approved items and fail closed to the
  existing empty state when the manifest or activity-manifest binding is
  invalid.
- Extract video completion event construction into a focused helper used by the
  runtime. It must keep the universal event shape, explicitly mark every skill
  outcome `completed`, and include machine-readable `exposure_only`, local
  source, manifest, media type, and duration metadata.
- Add focused tests with non-empty synthetic manifests so validation is not
  vacuous. Tests must prove a valid local item passes; external, mismatched,
  duplicate, unsupported, overlong, autoplay, loop, unapproved, and
  non-exposure items fail.
- Add evidence/mastery tests proving video completion produces completion
  evidence but zero counted attempts, zero successful transfer contexts, and no
  mastery promotion.
- Add a contract check that every real manifest's referenced local asset exists
  when items are later added.
- Advance package, baseline, and roadmap to v0.3.33 and identify the first
  approved local clip plus a separate response activity as the next content
  slice.

### Must Not Change

- `src/core/mastery-engine.ts`, `transfer-coverage.ts`, `skill-outcomes.ts`,
  `progress.ts`, and `adaptive-engine.ts`: their current counted-outcome and
  mastery rules already keep completion exposure out of mastery.
- The universal activity, event, storage, transfer, curriculum, and progress
  schemas. Manifest fields remain module-specific; exposure metadata uses the
  existing event metadata and per-skill outcome records.
- `curriculum.v1.json`, vocabulary thresholds, prerequisites, or transfer
  requirements.
- The empty child-facing Video Vault state, direct route, and absence from the
  four-tile home screen.
- Parent gate, approval, export, reset, observations, recommendations, and
  difficulty application behavior.
- Number Train, Spatial Scene, Coloring, Word game, Bear Cafe, and their tests or
  contracts.
- No actual video asset, thumbnail, follow-up vocabulary activity, new game,
  backend, auth, cloud sync, upload/import UI, IndexedDB migration, external
  content, dependency, autoplay, loop, reward, streak, timer, child-facing AI,
  or pressure.
- The unrelated untracked `.github/workflows/` directory.

### Acceptance Standard

- The real empty manifest validates under the repo-bundled exposure-only policy.
- Non-empty synthetic valid local media validates and is selectable by the
  runtime helper.
- Every declared boundary is tested with one passing and one failing side where
  applicable; malformed media fails closed.
- Activity/manifest id mismatch produces no playable media.
- Video completion events explicitly identify exposure-only evidence and local
  provenance while preserving the existing event contract.
- Completion exposure creates no counted vocabulary attempt, no accuracy or
  transfer evidence, and no mastery promotion.
- Existing child UI remains the same because no media is added.
- Parent approval and all external-content safety rules remain intact.
- Tests, typecheck, build, lint if present, and `git diff --check` pass.
