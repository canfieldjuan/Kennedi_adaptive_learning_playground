# v0.3.24 Symbolic Word-Building Transfer

## Before Code

### Code Read Cold

- `build-cat`, `build-dog`, and `build-sun` all use the existing Word Builder
  runtime with a picture, target-specific spoken prompt, blank slots, and
  shuffled letter tiles.
- Those activities declare only weak `same_format_same_examples` or
  `same_format_new_examples` transfer contexts with `prompt_mode: mixed`.
- The `word_building` curriculum plans `different_prompt_mode`, but the approved
  catalog contains no medium or strong word-building context.
- The content-gap engine recommends `build_word_from_spoken_prompt` / "Build the
  Word You Hear". The runtime uses `content.prompt_audio` for both visible prompt
  text and speech, so that proposal cannot be implemented honestly without
  either displaying the target word or making play depend entirely on speech.
- The runtime already supports an absent picture, but it has no visual task
  object that can present a symbolic model word.

### Root Cause

Word-building transfer is blocked by a missing prompt modality and an
unimplementable brief. The evidence engine can ask for a different prompt mode,
but every shipped activity uses the same picture-plus-spoken format, while the
recommended spoken-only variant conflicts with the runtime's shared visible and
spoken prompt and the product's no-audio-only safety boundary.

### Correct Fix Must Touch

- `src/modules/phonics-match/WordBuilderActivity.ts` must render one optional
  symbolic word-model task object when activity content provides `word_model`.
  Existing picture activities must retain their current rendering and events.
- Word Builder CSS must style that model as a compact, readable learning object
  with stable dimensions and mobile-safe sizing; it must not shrink existing
  touch targets or add layout clutter.
- One new approved activity must ask the child to copy a printed model word by
  tapping shuffled letters in the existing runtime. It must declare
  `different_prompt_mode`, `prompt_mode: symbolic`, cite
  `brief-word_building-different_prompt_mode`, omit picture content, and remain
  outside the fixed child chain.
- The activity catalog must register the variant so parent transfer
  recommendation can find and launch it.
- The content-gap engine must recommend the symbolic activity actually
  implementable by this slice instead of the unsafe spoken-only brief.
- Rich-transfer truth validation must require model/target agreement, exact
  buildable tiles, no picture, a generic non-answer-disclosing prompt, and the
  existing tap-then-place interaction.
- Focused tests must prove rich-content truth, weak-to-medium recommendation,
  ordinary attempt/completion events, symbolic model rendering, existing
  picture rendering, and unchanged fixed-chain order.
- Version and baseline/roadmap documents must identify v0.3.24 and its remaining
  transfer gaps.

### Must Not Change

- `src/core/mastery-engine.ts`, `transfer-coverage.ts`, `evidence.ts`,
  `progress.ts`, and `adaptive-engine.ts`: no mastery, context-strength,
  retention, promotion, or evidence-threshold changes.
- Activity, event, storage, and curriculum schemas: `content` is already an open
  record, so this slice needs no migration or new universal field.
- The fixed Word-game `next_activity_id` chain: symbolic transfer is
  parent-launchable, never automatic child routing.
- Existing build activity JSON, pictures, prompts, difficulty, versions,
  feedback, and events.
- Speech service, voice selection, repeat behavior, parent gate, parent
  approval, recommendation application, and local storage behavior.
- Initial-sound, blending, Bear Cafe, Math, Art, Spatial, Video Vault, home
  layout, and unrelated child UI.
- No new game, reward, streak, timer, backend, auth, cloud sync, external
  content, dependency, child-facing AI, or pressure language.
- The unrelated untracked `.github/workflows/` directory.

### Acceptance Standard

- The new activity renders a visible symbolic model word, no picture, three
  large shuffled letter tiles, and the existing blank build slots.
- The model word equals the target and success-rule target; the tiles spell that
  word exactly and the prompt does not disclose the target outside the model.
- Successful weak picture-building evidence makes coverage ready for transfer
  and recommends the symbolic activity; no launch or mastery is automatic.
- Correct completion emits the existing correct and completed event shape for
  `word_building`.
- Existing picture builders and the fixed child chain are unchanged.
- Parent approval and all safety boundaries remain intact.
- Tests, typecheck, build, lint if present, `git diff --check`, and narrow mobile
  browser QA pass.
