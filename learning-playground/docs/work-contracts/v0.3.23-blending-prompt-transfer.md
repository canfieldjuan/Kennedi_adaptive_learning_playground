# v0.3.23 Blending Prompt-Mode Truth and Transfer

## Before Code

### Code Read Cold

- `blend-cat`, `blend-hat`, and `blend-bat` render a visible sound strip from
  `content.segments`. Their spoken prompts give only a generic instruction and
  do not speak the individual phonemes.
- Those same activities declare `transfer.prompt_mode: spoken`, so their
  metadata does not describe the prompt that the child actually receives.
- All approved blending activities share the weak
  `same_format_same_examples` context and the `tap-picture-blend` context id.
  The catalog therefore has no implemented medium or strong blending transfer
  activity.
- The v0.3.22 content-gap engine now correctly recommends
  `different_prompt_mode` and the existing Word game for a blending gap, but no
  approved activity can satisfy that recommendation.

### Root Cause

Blending transfer is blocked by both missing content and inaccurate metadata.
The engine can name the needed prompt-mode transfer, but the approved catalog
contains only visually presented sound strips whose metadata incorrectly calls
them spoken prompts. Adding another activity without first distinguishing the
existing visual prompt from a genuinely spoken-sound prompt would let the
mastery system count a false context difference.

### Correct Fix Must Touch

- Existing blending activity JSON must declare the visual prompt mode that the
  runtime actually renders. Every changed activity must receive a new version
  so historical event meaning remains stable.
- One new approved blending activity must use the existing phonics matcher to
  speak the phoneme sequence and ask the child to choose the blended word. It
  must declare `different_prompt_mode`, cite
  `brief-blending-different_prompt_mode`, remain outside the fixed child chain,
  and preserve visual answer choices plus repeat access.
- The approved activity catalog must register the new activity so parent
  transfer recommendation can find and launch it.
- Rich-transfer validation must recognize spoken blending content only when
  the activity truthfully carries a target word, spoken phoneme sequence,
  matching prompt, correct visual answer, and no visual sound-strip segments.
- Focused tests must prove metadata/version truth, rich-transfer validation,
  parent recommendation availability, ordinary attempt/completion events, and
  unchanged fixed-chain order.
- Version and baseline/roadmap documents must identify the landed v0.3.23
  capability and current verification count.

### Must Not Change

- `src/modules/phonics-match/PhonicsMatchActivity.ts` and
  `WordBuilderActivity.ts`: no runtime, feedback, hint, event, or child-layout
  behavior changes.
- The fixed Word-game `next_activity_id` chain: the richer transfer activity is
  parent-launchable evidence, not automatic child routing.
- `src/core/mastery-engine.ts`, `transfer-coverage.ts`, `evidence.ts`,
  `progress.ts`, and `adaptive-engine.ts`: no mastery, strength, promotion,
  retention, or evidence-threshold changes.
- Activity, event, storage, and curriculum schemas: no migration or new field.
- Initial-sound, letter-sound, word-building, Bear Cafe, Math, Art, Video Vault,
  mobile layout, voice selection, parent gate, and parent approval behavior.
- No new game, reward, streak, timer, backend, auth, cloud sync, open web
  content, external asset, dependency, or child-facing AI.
- The unrelated untracked `.github/workflows/` directory.

### Acceptance Standard

- Existing sound-strip blending activities declare `prompt_mode: visual` and
  have bumped versions.
- The new activity is an approved `blending` activity with medium
  `different_prompt_mode` transfer and an originating brief reference.
- The spoken prompt contains the declared phoneme sequence without saying the
  whole target word before the answer; the correct choice matches the target
  and the activity does not render the existing visual sound strip.
- Blending coverage can move from weak-only to an available medium transfer
  recommendation without granting mastery automatically.
- Parent approval remains required before launch, and the fixed child chain is
  unchanged.
- All tests, typecheck, build, lint if present, and `git diff --check` pass.
