# v0.3.37 Parent Observation Signals

## Before Code

### Code Read Cold

- `ParentObservation` stores only ids, a free-text note, and timestamps.
- Storage validates only those legacy fields. Export and reset already include
  the full local observation array.
- The Parent Panel provides one textarea and saves every entry without a fit
  category or skill scope. History shows only timestamp and note.
- Parent interpretation searches every session note for frustration keywords for
  every reviewed skill. A counting-specific frustration note therefore affects
  unrelated skills touched in the same session.
- Mastery evidence treats any observation whose note contains a skill id or
  label as positive `parent_observation` evidence. Sentiment is ignored, so a
  note such as “Vocabulary was frustrating” can strengthen evidence.
- Existing observations must remain readable, reviewable, exportable, and
  clearable without a storage migration.

### Root Cause

Parent observations have no machine-readable meaning or scope. The app infers
both from unrestricted prose, which makes interpretation imprecise and can turn
difficulty/frustration language into positive evidence. This is a modeling
error, not a copy or display problem.

### Correct Fix Must Touch

- Add an optional, backward-compatible observation category and optional
  `skill_ids` to `ParentObservation`.
- Define a small closed category set for general note, independent success,
  needed support, too easy, about right, too hard, frustration, and real-world
  transfer, with parent-readable labels and shared semantic helpers.
- Extend storage validation so old observations remain valid while malformed
  new categories or skill scopes fail closed.
- Add category and skill selectors to the parent-only notes editor. New notes
  save the selected category and zero or one session skill; the note remains
  required.
- Show category and optional friendly skill labels in note history.
- Scope structured support signals to their tagged skill. Unscoped observations
  remain session-wide. Preserve keyword fallback for legacy/general notes.
- Only `independent_success` and `real_world_transfer` structured observations
  may create parent mastery evidence, and only for explicitly tagged skills.
  Structured difficulty, support, fit, frustration, and general categories must
  not become positive mastery evidence.
- Preserve legacy note-to-skill evidence matching for pre-v0.3.37 observations.
- Keep export/reset behavior unchanged so additive fields flow through existing
  local export without a migration or second storage key.
- Add focused storage, history, evidence, interpretation, and parent-UI contract
  tests.
- Advance package, baseline, and roadmap to v0.3.37.

### Must Not Change

- Mastery statuses, thresholds, confidence math, transfer coverage/strength,
  review scheduling, curriculum, recommendations, or automatic difficulty
  application beyond consuming correctly scoped support observations.
- Activity/event schemas, event logging, progress profiles, storage keys, export
  version, or clear-data ownership.
- Parent gate, transfer/brief decisions, mastery snapshots, review records,
  active guidance, or difficulty override controls.
- Child routes, home screen, child CSS, Video Vault, Number Train, Bear Cafe,
  Words, Art, spatial activities, or any game behavior/content.
- No backend, auth, cloud sync, IndexedDB migration, analytics, external content,
  child-facing observation labels, automatic routing, rewards, streaks, timers,
  pressure language, or dependency changes.
- The unrelated untracked `.github/workflows/` directory.

### Acceptance Standard

- Existing observations with no category or skill ids still load, export, show,
  and use legacy note matching.
- New observations round-trip valid categories and skill ids through local
  storage/export; invalid categories and malformed skill scopes are filtered.
- Parent Notes offers a category menu and a session-skill menu without changing
  child-facing UI.
- History shows a friendly category and optional friendly skill label.
- A counting-tagged frustration/support observation affects counting only, not
  another skill in the same session.
- Structured independent success or real-world transfer can cite its observation
  id as evidence for a tagged skill even when the prose omits the skill name.
- Structured too easy/about right/too hard/support/frustration/general notes
  never become positive mastery evidence.
- No structured observation grants mastery by itself or changes mastery rules.
- Tests, typecheck, build, lint if present, `git diff --check`, and parent-panel
  browser QA pass.

## Review Finding Verification

### Gaps First

- Contradicted cause: the review cites `increase_difficulty` producing `Promote
  gently`, but `src/core/mastery-engine.ts:235` maps every current mastery status
  before the fallback at `src/core/mastery-engine.ts:257`. That concrete path is
  not reachable in the current status model.
- Confirmed existence and location: `getStatus` consumes a qualifying parent
  support signal at `src/core/parent-interpretation.ts:220`, but
  `getRecommendation` returns mapped mastery advice first at
  `src/core/parent-interpretation.ts:234`. A transfer-ready skill can therefore
  display `Needs more support` and `Try transfer activity` together.
- Confirmed secondary location: even if recommendation selection is corrected,
  `getRecommendationReason` returns the mastery explanation first at
  `src/core/parent-interpretation.ts:293`, so the displayed reason can still
  contradict the parent support decision.
- Confirmed severity: this is a P2 because a plausible high-evidence skill plus
  a tagged `too_hard`, `needed_support`, or `frustration` observation produces
  contradictory primary Parent Guidance.

### Corrective Contract

- When the session has enough attempts for parent support to affect status, the
  same skill-scoped support signal must return `Review later` before any mastery
  recommendation is mapped.
- A support-owned `Review later` recommendation must use the support explanation
  before any mastery explanation.
- Add a regression test with strong transfer-ready evidence and a tagged parent
  support observation; assert coherent status, recommendation, and reason.
- Do not change mastery statuses, mastery actions, transfer evaluation,
  low-evidence behavior, observation scope, or parent approval rules.
