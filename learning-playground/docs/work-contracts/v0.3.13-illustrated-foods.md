# v0.3.13 Illustrated Foods

Second surface of the illustrated art standard (issue #3), after the approved
illustrated bear. Applies the same flat, ink-outlined SVG style to the Bear Cafe
foods. Shapes are new art, so it carries a spot-check note (recognizability
matters in a learning game).

## Before Code

### Root Cause

After the illustrated bear (#22) the foods are still emoji glyphs
(`content.food.icon`, 🍌🍪🫐…), so the Bear Cafe is visually half-illustrated.

### Correct Fix Must Touch

- `src/modules/kennedis-orders/food-art.ts` (new) — `renderFoodArt(foodId)`
  returning local inline SVG for the seven foods (apple, banana, berry, bread,
  cookie, cupcake, soup); unknown ids fall back to a neutral plate.
- `src/modules/kennedis-orders/KennedisOrdersActivity.ts` — render the SVG at the
  food sites: choice buttons, the tray/plate items, and the count-expanded plate
  in `getPlatedFoodIcons` (used by the plating + handoff beats).
- `tests/modules/kennedis-orders.test.ts` — the plated-quantity regression now
  counts food SVGs instead of emoji.
- `src/styles/child-ui.css` — a `.food-art` sizing rule; remove the now-dead
  `.bear-cafe-reaction*` rules + `cafeReaction*` keyframes + guard entries
  (retired with the bear in #22).
- `package.json` / `package-lock.json` — version bump to `0.3.13`.
- Docs note in `MVP_BASELINE.md`.

### Must Not Change

- **Safety boundary:** SVG authored inline (no external assets/network/deps); no
  `http(s)`.
- Activity JSON / `content.foods` untouched (the emoji `icon` stays in the data;
  the runtime renders the SVG) — no schema change, trivially revertible.
- **Count integrity preserved:** `getPlatedFoodIcons` still expands by quantity
  (the #16 P2), now verified by SVG count in the regression test.
- No change to the evidence model, mastery, transfer, storage, parent panel,
  home grid, or the child gate. Decorations (bubbles/hearts/sprinkles/stars) are
  a separate render path and stay emoji this slice (scene follow-up).
