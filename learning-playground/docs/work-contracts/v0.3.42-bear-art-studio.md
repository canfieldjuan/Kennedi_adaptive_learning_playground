# v0.3.42 Bear Art Studio (art arc slice 1)

First real version of the Art game: the child is the artist for the Bear
family. Five progression modes on one new narrow runtime, mapped entirely to
existing curriculum skills.

## Before Code

### Root Cause (slice)

The art lane is a one-shot color-fill stub. `ColoringActivity` proves a fill
renders and emits events (one swatch → one circle), but it offers no fantasy
(no requester, no reason to make art), no progression (both activities are
single fills), no replay depth, and no strong transfer contexts for a child
who is past baby-level color tapping. The Art home tile is the only game
tile that still opens a worksheet instead of a game.

### Required Change Surface

- `src/modules/bear-art-studio/` (new) —
  - `BearArtStudioActivity.ts`: the runtime. One screen: utility strip,
    bear requester portrait (reusing `kennedis-orders/bear-art`'s three
    characters + waiting/receiving/happy faces), an icon-first visual
    request card, the art board, a color/sticker tray, and mode-appropriate
    Done/Check controls. Five `content.art_mode`s:
    - `free_decorate` — pick colors/stickers, apply to slots, Done when at
      least one decoration is placed; no wrong answers.
    - `color_request` — paint the subject the requested color; wrong color
      = gentle incorrect (stays correctable), 2 misses = hint glow.
    - `quantity_decorate` — add/remove stickers on slots; evaluated ONLY on
      Check (Number Train's honesty rule: never per tap); over/under-count
      recorded; bounded count-along hint; never auto-filled.
    - `pattern` — a scarf strip with the pattern's start pre-filled; each
      swatch tap fills the next segment; wrong-for-position = gentle
      incorrect + wiggle; completing the sequence completes the piece.
    - `fix_art` — the board arrives with exactly one region painted wrong
      vs the request card; repainting that region with the requested color
      fixes it; tapping an already-matching region = gentle redirect.
  - `studio-art.ts`: local inline sticker/subject SVGs in the arc standard
    (star, heart, flower, sun, bubble; scarf/card surfaces) + render-check
    export. No `<text>`, no emoji.
  - `bear-art-studio.types.ts`: content parsing types.
- Routing: `src/app/main.ts` dispatches `interaction_model:
  'tap_then_place'` + `content.game === 'bear-art-studio'` to the new
  runtime (the exact Bear Cafe / Number Train key pattern; no schema
  change) + destroy wiring.
- `src/modules/home/HomeScreen.ts` — the Art tile routes to the studio
  chain entry (`art-studio-free-decorate`), the same switch Math made when
  Number Train shipped. Structure/labels/speech untouched.
- `src/content/activities/` (5 new): `art-studio-free-decorate.json` (d0,
  color_fill, same_format_new_examples), `art-studio-pink-request.json`
  (d2, color_fill, different_prompt_mode), `art-studio-three-stars.json`
  (d1, counting, different_interaction_model),
  `art-studio-pattern-scarf.json` (d4, color_fill — L2 "uses color in
  combined orders" — different_prompt_mode), `art-studio-fix-card.json`
  (d3, color_fill, different_prompt_mode). Chained by `next_activity_id`
  (free → pink → stars → pattern → fix → Home).
- `src/content/activity-catalog.ts` — register the five.
- `src/content/curriculum/curriculum.v1.json` — DATA ONLY: append the new
  activity ids to `color_fill.activity_contexts` and
  `counting.activity_contexts`. **No new skills, no new levels, no band
  changes** (pattern maps to color_fill level 2; quantity maps to counting
  level 0 band).
- `src/styles/child-ui.css` — `.bear-art-studio` block. The container also
  carries `coloring-studio`, reusing the shipped Art-studio scene
  (`createStudioEnvironment`) + its ink/text overrides; new styles cover
  requester strip, request card, board/slots, sticker tray, scarf
  segments, and the phone layout (top strip / board / bottom tray — no
  sad-form stacking, no scrolling during play, targets ≥ 56px).
- Tests — `tests/modules/bear-art-studio.test.ts` (behavioral, all five
  modes: completion only on legitimate finish, check-only quantity
  evaluation with over/under metadata, requested/selected color metadata,
  pattern position metadata, fix redirect, hint flows, chain button, scene
  inert). Contract suites (schema validation, safety, curriculum skill
  existence, transfer metadata) pick the five activities up automatically
  via the catalog.
- `docs/contracts/art-studio.contract.md` (new, NOT registered in
  `scripts/check-work-contract.mjs`) — the art-lane product contract.
- `package.json` / `package-lock.json` — `0.3.41`; `MVP_BASELINE.md` note.

### Explicit Non-Scope / Must Not Change

- `ColoringActivity` and the two existing art activities — untouched and
  still registered (direct route + parent lane; transfer-coverage tests
  reference them).
- Mastery rules, transfer strength rules, parent approval rules, progress
  derivation, event consumers — untouched. One event system
  (`ActivityAttemptEvent`), no second progress store.
- Curriculum graph structure: no new skills/levels/prerequisites/unlocks
  (data-only activity_contexts additions).
- No backend/auth/cloud/AI/generative anything, no external or remote
  assets, no free-drawing canvas, no gallery persistence, no reward
  economy/streaks/leaderboards/sharing, no schema changes, no dependency
  changes, no broad refactors.
- No harsh failure states anywhere: incorrect = gentle wiggle + supportive
  line; no red-X feedback; completion is the only ending.

### Assumptions / Blockers

- `tap_then_place` + game key routing is the established extension point —
  no `interaction_model` enum change needed (verified against schema).
- Reusing `renderBearArt` (kennedis-orders) and `createStudioEnvironment`
  (coloring-book) across modules follows the home-icons precedent.
- The Art tile retarget is safe: no test pins the art route (verified by
  grep; home tests pin structure/labels/Words/Cafe routes only).
- Difficulty levels chosen sit inside existing skill bands (color_fill
  0-1/2-4/4-5, counting 0-1) — no coverage validation change.
- Blockers: none known.

## Contract Amendments

- **Version renumbered 0.3.41 → 0.3.42.** Codex's browser-viewport PR (#63)
  merged as 0.3.41 mid-slice (also adding `@playwright/test`); origin/main
  merged in, slice renumbered (routine collision procedure).
- **Transfer-honesty detectors added to `validate-activities.test.ts`.**
  The contract test requires every rich transfer claim to match detectable
  content. Three detectors were added following the `isKennedisOrdersFixOrder`
  precedent: the studio's visual color request (color present exactly once
  in the palette and never named in speech), the fix-card comparison
  (exactly one wrongly-painted region, intended color never spoken), and —
  the first use of `different_interaction_model` on an implemented
  activity — the sticker-quantity counting mode (judged only on Check).
  The strength mapping itself (`TRANSFER_CONTEXT_STRENGTHS`) is untouched.
- **Two consumer tests updated to the new content reality.**
  `parent-interpretation.test.ts` asserted counting's
  `different_interaction_model` context was *missing*; the studio now
  provides it, so the assertion flipped (only `delayed_review` remains
  missing) — the engine behaving exactly as designed.
  `number-train.test.ts`'s home-route pin updated for the Art tile's
  intended retarget (same switch Math made when Number Train shipped).
