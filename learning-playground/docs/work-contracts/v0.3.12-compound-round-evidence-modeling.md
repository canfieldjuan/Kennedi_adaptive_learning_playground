# v0.3.12 Compound-Round Evidence Modeling

## Before Code

### Root Cause

Bear Cafe compound rounds can exercise more than one skill in one order, but the
local event model previously had one overall `outcome` for the whole activity
attempt. Progress and mastery evidence grouped that same event under every
listed skill and then scored each skill from the global outcome. A child who
counted the correct quantity but chose the wrong color therefore produced
incorrect counting evidence, and a child who chose the correct color but missed
quantity produced incorrect color evidence.

### Correct Fix Must Touch

- `src/types/events.ts`
  - Add an optional per-skill outcome list to attempt events without removing or
    changing the overall activity outcome.
- `src/core/event-log.ts`
  - Validate and preserve the optional per-skill outcome field while keeping
    legacy localStorage events readable.
- `src/core/progress.ts` and `src/core/evidence.ts`
  - Prefer per-skill outcomes when present and fall back to the global outcome
    when old events do not have per-skill outcomes.
- `src/modules/kennedis-orders/KennedisOrdersActivity.ts`
  - Emit per-skill outcomes for the existing two-part Bear Cafe order.
- Focused tests for logger preservation, progress scoring, evidence scoring, and
  Bear Cafe event emission.
- Version/baseline/current-status docs for v0.3.12.

### Must Not Change

- Do not add new game modules, new child-facing activities, backend services,
  auth, cloud sync, analytics, rewards, streaks, rankings, child-facing AI, or
  open web content.
- Do not change the Bear Cafe child flow, activity chain, home grid, parent
  approval rules, reset/export section names, safety rules, transfer rules,
  curriculum ladders, or recommendation application behavior.
- Do not require a schema-breaking localStorage migration; old events must still
  score from the global outcome.

## Cold Diff Audit

### Gaps

### Change By Change Reconstruction

- `events.ts` adds optional `skill_outcomes` rows with `skill_id`, `outcome`,
  and optional `reason`, leaving the overall event `outcome` intact.
- `event-log.ts` validates `skill_outcomes` when present, preserves valid rows,
  and keeps old events readable through the existing normalization path.
- `skill-outcomes.ts` centralizes the rule: explicit per-skill rows win, missing
  rows mean no skill-specific outcome for that event, and events without rows
  fall back to their global outcome.
- `progress.ts` groups and scores events with the shared skill-outcome rule,
  including promotion, needs-review, abandon, and hint calculations.
- `evidence.ts` derives counted attempts, correct attempts, completions, hints,
  and transfer contexts from the shared skill-outcome rule.
- `KennedisOrdersActivity.ts` emits per-skill quantity/color outcomes for the
  two-part order and attaches hint events only to the skill being hinted.
- Tests cover logger validation/export, progress scoring and legacy fallback,
  evidence scoring, and Bear Cafe two-part event emission.
- `package.json` / `package-lock.json` bump to `0.3.12`.
- `MVP_BASELINE.md` and `CURRENT_STATUS_AND_ROADMAP.md` record the new baseline.

### Contract Traceability

- Compound-round scoring now reaches the cause: per-skill evidence can differ
  inside one activity while the whole order can still be incorrect overall.
- Legacy event fallback is preserved, so this is additive and does not require a
  storage migration.
- Child gameplay, content breadth, parent approval, safety, backend posture,
  transfer policy, curriculum ladders, and recommendation application remain
  outside this change.
