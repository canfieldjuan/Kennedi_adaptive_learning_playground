# v0.3.27 Visual Color Request Transfer

## Before Code

### Code Read Cold

- `art-color-circle` and `art-color-circle-cool-colors` use the same
  `color_fill` runtime, shape, and free-choice interaction. Their transfer
  contexts are weak `same_format_same_examples` and
  `same_format_new_examples` with `prompt_mode: mixed`.
- `ColoringActivity` completes after any selected swatch is applied and records
  that selected color as the correct answer. It has no target-color or
  incorrect-selection path.
- The `color_fill` curriculum plans `different_prompt_mode`, but the approved
  catalog contains no medium or strong Art transfer activity.
- The content-gap engine displays the Art pattern "Color From Request Card" but
  returns the generic `same_color_skill_new_shape` template, so its machine-
  readable and parent-readable recommendations disagree.
- No behavioral test directly exercises `ColoringActivity`.

### Root Cause

Art transfer is blocked by a free-choice-only runtime and contradictory content
guidance. The engine can name a visual request-card context, but the runtime
cannot distinguish matching a requested color from choosing any color. Calling
a clone medium transfer would let incorrect selections complete and would turn
free choice into false evidence of color matching.

### Correct Fix Must Touch

- `src/modules/coloring-book/ColoringActivity.ts` must support an optional
  `target_color_id` drawn from the existing palette. Without it, current
  free-choice behavior and event meaning must remain unchanged.
- In target mode, the runtime must render a visual request card, keep the same
  large swatches and fill target, record a wrong applied color as incorrect,
  keep the round playable, and complete only when the selected color matches.
- Repeated mismatch must show one bounded visual/verbal hint, highlight the
  requested swatch, and emit the existing hint event shape.
- Event records in target mode must distinguish selected and correct color ids,
  labels, response time, attempt number, hint use, and target metadata without
  changing the universal event schema.
- Scoped coloring CSS must style the request card, mismatch, hint, mobile fit,
  and reduced-motion behavior without shrinking existing targets.
- One new approved activity must declare `different_prompt_mode`, visual prompt
  mode, a target color contained exactly once in its palette, and
  `brief-color_fill-different_prompt_mode`. It must remain parent-launchable and
  outside child home routing.
- The activity catalog and content-gap engine must expose the activity and a
  matching visual-request template.
- Focused tests must prove legacy free choice, target mismatch, correction,
  bounded hint, fail-closed invalid target, truth validation, transfer
  recommendation, parent approval, and no external content.
- Version, baseline, and roadmap documents must identify v0.3.27 and remaining
  spatial/video transfer gaps.

### Must Not Change

- `src/core/mastery-engine.ts`, `transfer-coverage.ts`, `evidence.ts`,
  `progress.ts`, and `adaptive-engine.ts`: no mastery, strength, retention,
  promotion, or threshold changes.
- Activity, event, storage, and curriculum schemas: target data belongs in the
  existing open activity content/success records.
- Existing Art activity JSON, palette, free-choice prompt, versions, and
  completion behavior.
- Home routing and the four child tiles: the transfer activity is launched only
  through the existing parent-controlled recommendation path.
- Number Train, Word game, Bear Cafe, Spatial, Video Vault, speech/voice,
  parent gate, parent approval, and persistence behavior.
- No new game, reward, streak, timer, backend, auth, cloud sync, external
  content, dependency, child-facing AI, reading requirement, or pressure.
- The unrelated untracked `.github/workflows/` directory.

### Acceptance Standard

- A legacy free-choice round still completes with any selected palette color
  and records the selected color as its correct answer.
- A visual-request round shows the target color without requiring its name to
  be read or heard.
- A wrong applied color emits incorrect evidence, does not complete, and can be
  corrected.
- The bounded hint appears once after the configured mismatch threshold and
  highlights the matching swatch.
- A correct match emits correct and completed events with selected/correct
  color agreement and target metadata.
- Missing or unknown target-color configuration fails closed.
- Weak Art evidence makes transfer ready and recommends the approved visual
  request activity; no launch or mastery is automatic.
- Parent approval and all safety boundaries remain intact.
- Tests, typecheck, build, lint if present, `git diff --check`, and narrow mobile
  browser QA pass.
